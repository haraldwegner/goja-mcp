# GOJA v2.6.1 — find_quality_issue sweep hardening

A patch for three defects in `find_quality_issue` surfaced by dogfooding v2.6.0 on a large,
git-tracked, deployed project (across both Claude Code and Cursor). All three are invisible
to single-file fixture tests — they only manifest at real scale — and are now guarded by
reproduction tests.

- **Bounded family sweeps.** A whole-family sweep on a large repo returned thousands of
  findings (200k+ chars) — unconsumable inline by an MCP client. `find_quality_issue(family=…)`
  now supports:
  - `summary: true` — returns counts-by-kind + the `conflicts` list only (no full findings
    array): the right shape for a broad sweep;
  - a default findings cap with `offset` / `limit` paging. The full total is always in `count`;
    pagination is applied *after* baseline diffing, so trend diffs still see the full set.

- **`filePath` now scopes the churn detectors.** `divergent_change` (and the git-history
  family) ignored `filePath` and scanned the whole project, so a single-file scoped sweep
  leaked project-wide findings. It now honours `filePath` like every other detector.

- **Consistent project-relative paths.** A relative `filePath` was resolved against the process
  CWD (an AppImage mount on a deployed resident), leaking an absolute mount path into findings
  and fragmenting the multi-catalog conflict seam (AST findings absolute, churn findings
  relative → never co-located). All detectors now resolve `filePath` against the project root
  and emit a consistent project-relative path.

No API breaks; new params (`summary` / `offset` / `limit`) are additive. goja-studio is
unchanged (stays at v2.6.0).
