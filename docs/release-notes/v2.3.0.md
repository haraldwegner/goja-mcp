# goja-mcp v2.3.0 — Looping atomic facts (Sprint 21c, items A + B)

The store's unit becomes the atomic FACT, and the keyword surface becomes hundreds —
"otherwise the memory store is just another way to store the md entries."

## Loader v4: keyword harvest (item A)

Memory-file bodies are harvested for their cue-dense structure — headings (all
levels), `**bold**` phrases, `` `backticked` `` terms and `[[wikilink]]` names become
symptom rows the recall matcher can actually hit. Fenced ``` code blocks are skipped
(code is not a cue source). Harvested phrases are deduplicated after normalization,
truncated to the symptom column, and capped at 30/entry as a reported backstop
(`keyword_capped` in the load report).

## Section-splitting: one entry per fact (item B)

Files are bundles; the fit gate should answer with the fact. Bodies split at heading
boundaries (any level, fence-aware): a thin file-level parent (frontmatter summary +
preamble details) plus one entry per section — summary = the heading, details = the
section body, symptoms = that section's keyword harvest, links = the section's own
wikilinks, `scope_kind = section`. The whole family shares the file-level
`source_ref` + `source_hash`: skip-unchanged and delete-by-source are untouched, and
a changed file re-ingests as one unit. Headingless files keep their existing shape.

## Recall answers with the fact, not the bundle

When a section and its file-parent both fit a cue, the parent is dropped from the fit
set. Sibling sections that both fit remain visible — the injection boundary (the new
goja-studio `UserPromptSubmit` hook, v2.3.0) enforces single-fact-or-nothing on top.
Terminal-or-absence is preserved throughout: never a pile, never a guess.

## Clean-up safety

`dedup` no longer touches file-backed entries at all: families are idempotent (the
next load resets them) and generic headings ("Context", "DoD") repeat across files —
the clean-up chain must never eat sections. Files are the source of truth; dedup's
domain is recorded entries.

## One-time corpus self-heal

`LOADER_VERSION` 3 → 4: on the first load after upgrading, every stored hash is stale
by definition and the whole corpus re-ingests once — splitting into facts and gaining
the keyword surface retroactively, with no manual steps. Subsequent loads are
byte-strict skips again. (This release also carries the loader-v3 fix: HTML-comment
managed-section markers are never derived summaries.)

No schema migration; no tool-surface change (the `experience` tool keeps its 15 kinds).
