# jawata-mcp v3.2.0 — the intelligent injector (Sprint 26)

The resident stops being a passive answerer and starts LEARNING from normal
use: every tool outcome is a label, every hand edit gets its consequence
recorded, and the hand rules that steer agents today become the baseline a
model must beat before it is allowed to decide anything. Training is a side
effect of work — there is no training step. The tool surface is unchanged:
**toolCount stays 45**; everything below rides existing tools as new kinds.

## Session identity + the event tap (D7's wiring)

Every call now carries an MCP session: `Mcp-Session-Id` is honored when the
client sends it, minted (`http-<uuid>`) when it doesn't, and echoed in every
response; stdio gets one session per process. At the registry choke point —
the one place every tool outcome converges — the EVENT TAP turns outcomes
into persisted learner events (H2 schema v5: `learner_event`,
`learner_state`): tool errors, undos, mechanical touches, gate calls
(compile/diagnostics/tests), and the compile-after-touch failure label. A
failed event write is LOUD (counted and reported), never silent.

## The automatic architect — watch injection (D1)

A change you make is reviewed WITHOUT being asked: the disk-sync delta plus
each call's modified files run through single-file detectors (never a family
sweep), diff against a retained per-file baseline, and NEW findings ride the
next tool answer as an appended steering block ending in the architect's
question — "design fix or bandage?". Noise budget 2 per answer, 30-minute
cooldown per finding, store-decay (a declined, unchanged finding goes
silent). Every finding is journaled; steering COMPOSES — a tool's own
steering line is never overwritten.

## The learners (D2) — continuous, gated, honest

Seven learners (edit switch, seat switch, policy dial, architect pre-filter,
noise budget, watch-firing, message-gating) share one machinery: a
hand-rolled online logistic regression updated PER EVENT plus a small
decision tree rebuilt every 25 events. Features are COMPILER FACTS ONLY
(AST-shape deltas — method/type counts, signature shape, control flow); a
test pins that no feature derives from raw text, so two edits differing only
in names produce identical vectors. Each learner keeps a rolling record
against the hand rule ON THE SAME EVENTS and DECIDES only while it beats the
rule on BOTH error kinds over the window — ties never promote, decay
demotes. Serving has a hard 50ms budget; over budget, the rule answers and
the model is advisory. Per the sprint's recorded risk decisions (R1/R2):
a model losing to the rules is a valid shipped outcome, and at release
every learner is honestly in SHADOW — the pipeline ships with measured
numbers, the rules keep deciding until the data says otherwise.

## The consequence-labeled edit feed (C7)

`experience(kind=observe_edit)` feeds the edit switch from real edits with
their TRUE outcome, not the rule's opinion: an unlabeled post pends in its
session until the next gate outcome (clean → trivial-ok, errors or an undo →
structural-mishandled); a labeled post (`outcome=clean|failed`, used by the
hook path, which correlates in the client session) trains immediately. The
end-to-end proof is in the suite: a scripted normal session advances every
counter and the policy dial with zero learning actions, live-verified over
real HTTP against the booted dist.

## Server-side enforcement lane (D3/D4/D5) — every client, no hooks needed

Riding the answers every client already reads:

- **Seat discipline (D4):** a seat outcome recorded in a session whose
  ledger shows no gate call is flagged ("a gate you did not run has NOT
  passed"); a seat-shaped session evicted without an outcome is journaled
  unjournaled. Mode-agnostic by construction — a deployed command and a
  rule-block stance look identical to the ledger.
- **The decision test (D5):** all three message shapes — decision ask,
  checkpoint summary, sprint result — detected in store records or in the
  call pattern; without audit evidence in the session, the DECISION TEST is
  attached to the answer.
- **The defect rule (D3):** a declared fallback AFTER a jawata tool error
  in the same session emits a RELAX label AND auto-files a defect record —
  both, always; silently learning around our own bugs is forbidden. The
  per-situation POLICY DIAL (BLOCK/STEER/SILENT) tightens on slips, loosens
  on legitimate relaxes, persists across restarts.

## Memory beyond Java (D6) + the client surface

Recall cues and the primer are de-Java-tuned: business-domain facts recorded
in one client are recallable by plain keyword in another and appear in a
fresh session's primer — no Java symbol required. The learning layer's
client surface is `experience(kind=train)` (forced pass + report) and
`(kind=learner_status)` (honest counters: events by kind, failed writes,
per-learner DECIDES/SHADOW with the rolling record) — used by the generated
`/train` command; `/memorize` rides `kind=record` store-first.
