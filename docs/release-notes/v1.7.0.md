# goja-mcp v1.7.0 — Result steering + error hints (Sprint 22, POST layer · tool half)

The tool half of the POST layer. After the PRE layer guides the agent to GOJA and enforces
try-first, v1.7.0 makes every result the agent reads *pay off* by naming the next grounded step —
reinforcing "stay in GOJA" from inside the response itself. **Zero net new tools** (surface stays
**40**). Ships paired with **goja-studio v1.5.0** (the PostToolUse observer).

## Result steering — every success names the next grounded step

`ResponseMeta.steering`, injected centrally at the dispatch choke point (`ToolRegistry.callTool`) so
it rides every successful result:

- after a navigate/understand tool (`search_symbols`, `find_references`, `analyze`, `inspect`, …) →
  "change what you found with a GOJA refactor tool (`rename_symbol` / `extract` / `move` /
  `refactoring(action=plan)`) — not a hand-edit";
- after a mutating tool → "verify with `compile_workspace` + `get_diagnostics`; GOJA changes are
  reversible (`undo` / `undo_plan`)";
- session/build/verify tools get none (no noise).

It's distinct from `suggestedNextTools` (tool-specific next *lookups*): steering names the
do → change → verify *direction*, reinforcing the PRE edit-gate from inside the result.

## Error hints — near-miss recovery

`ErrorInfo.invalidParameter` now carries a recovery hint (it was empty) → point at the tool's
`inputSchema`. (`symbolNotFound` already nudges `search_symbols`.)

## Compatibility

Additive: an optional `meta.steering` on success results + a hint on `invalidParameter` errors
(both `NON_NULL`-omitted otherwise). Every tool and action is unchanged; tool count remains **40**.
