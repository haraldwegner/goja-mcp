# goja-mcp v2.1.0 — Knowledge-store durability (Sprint 21a, Wave 1)

v2.0 shipped the experience store; v2.1.0 makes it **durable**. Two P0 fixes surfaced by the
v2.0 dogfood review, shipped ahead of the rest of Sprint 21a because data loss doesn't wait.

## Store continuity — your knowledge now survives restarts and redeploys

In v2.0 the store was opened inside the launcher's **session-isolation directory** — a random
subdir created per resident start and **deleted on clean shutdown**. Recorded knowledge was
orphaned on every redeploy and destroyed on every clean restart.

- **User-level shared store is the new default:** `${XDG_DATA_HOME:-~/.local/share}/goja/`
  `experience.mv.db`. Knowledge belongs to the user, not the workspace — methodology and
  domain entries are recallable from *every* workspace, while symbol/package scope-containment
  keeps repo-specific recall self-scoped exactly as before.
- **AUTO_SERVER:** multiple residents (different workspaces, or concurrent sessions of one)
  attach to the same store transparently.
- **Modes** via `-Dgoja.experience.store`: `shared` (default) · `workspace` (per-workspace
  file at the stable workspace root — never the session dir) · `memory` · an explicit
  directory path. The launcher now publishes the original `-data` path
  (`goja.workspace.root`) so the stable root is authoritative.
- **One-time recovery:** on startup the resident sweeps existing orphaned session stores and
  imports their entries (newest first, deduplicated, symptoms/links preserved, idempotent via
  markers; source files are never deleted).

## Versioned schema + migrations — the store can now evolve

v2.0's `CREATE TABLE IF NOT EXISTS` could create a store but never change one.

- **`schema_version` + ordered, additive-first migrations**; a real migration writes an H2
  online backup zip beside the store first.
- **Migration v2 adds facets:** `workspace_id` / `project_id` (provenance, stamped from
  `workspace.json` on every write) and `language` (default `java`) — the groundwork for the
  shared store and for multi-language safety (Wave 2).
- **From-the-future guard:** a store written by a newer resident is refused read-write with a
  clear message — and the server keeps running on a non-persistent in-memory store instead of
  dying.

## Notes

- Existing per-workspace v2.0 stores are picked up by the recovery sweep on the first v2.1
  start; nothing to do manually.
- Tool surface unchanged (41 tools; the `experience` front door gains no new kinds in Wave 1).
