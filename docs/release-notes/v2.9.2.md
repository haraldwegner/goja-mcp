# jawata v2.9.2 — dogfood patch on v2.9.1 (resident classpath + observer hook)

Dogfooding the 2.9.1 fleet found the classpath half of the self-workspace fix
regressing **in the resident only**: a Studio-launched process inherits a
desktop/AppImage PATH without the user's shell profile, `mvn` was not on it,
and the loader's classpath probe failed silently — 17,665 unresolved-import
errors while the identical code passed in a shell-launched sandbox.

## D4 — robust Maven executable resolution

The project loader now resolves Maven in this order instead of assuming `mvn`
is on PATH: the project's own wrapper (`mvnw`/`mvnw.cmd`) → PATH →
`MAVEN_HOME/bin` → well-known install locations (`/opt/apache-maven/bin`,
`/opt/maven/bin`, `/usr/share/maven/bin`, `/usr/local/bin`, SDKMAN). When
nothing is found it warns loudly instead of dying silently. Verified in a
sandbox with the resident's stripped PATH: 471 sources, 68 classpath entries,
0 compile errors.

## Observer hook (studio-managed, fixed on deploy)

- the fallback-reason extractor still matched the pre-rebrand marker — reasons
  recorded as empty junk (v2.9.1 notes; template fix ships with studio);
- NEW: the slip gate missed search commands **starting a line** — the request
  JSON carries newlines as literal `\n`, so `grep` was preceded by the letter
  `n` and failed the word-boundary check; those slips were silently never
  recorded. Gate checks now run on an escape-normalized copy.

## Tests

4 new tests on the `resolveMavenCommand` seam (wrapper preference, PATH hit,
known-location fallback — the resident case — and loud null). Importer suite
54/54 in-framework.
