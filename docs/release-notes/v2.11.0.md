# jawata v2.11.0 — memory-first navigation: knowing beats searching

Sprint 24, phase 1 of three. An agent that already knows a symbol should go
straight to it — the way you hit Open Type (Shift+Ctrl+T) in Eclipse instead of
grepping — and when a remembered name turns out to be stale, the answer should
correct it rather than just fail.

The insight: **the fully-qualified name is an agent's stable memory key.** It
survives file moves and package reshuffles; a file position does not. This
release makes that key work everywhere.

Sprint 23's `fields` projection cut tokens *per row*; this cuts *searches per
task*. They compose.

## Navigate by name, everywhere

Every reading tool and every whole-symbol refactoring now accepts the name form
— `symbol="com.foo.Bar#method"` or `typeName="com.foo.Bar"` — instead of a
file/line/column triple:

- `get_call_hierarchy(outgoing)` and `analyze`'s method/control-flow/data-flow/
  change-impact/symbol kinds (the reading tools that still demanded a position);
- `rename_symbol`, `change_method_signature`, `inline(method)`, `move(class)`,
  `move_in_hierarchy`, `move_method`, `encapsulate_field`,
  `extract(interface|superclass)`, `generate` (all kinds), and the type- and
  method-targeted `refactor_to_pattern` kinds.

Refactorings that target a statement or expression **range** keep their
coordinates — `extract(method|variable|constant)`, `compose_method`, the
switch-targeted pattern kinds, anonymous-class conversion. A range has no name.
That boundary was audited tool by tool before a line was written, and both the
wiring and its tests enumerate against that audit.

Explicit coordinates always win when you supply both, so nothing that worked
before behaves differently.

## Stale memory repairs itself

A name that no longer resolves now answers with the correction in the **same
response** — the whole human loop (look → miss → find → re-memorize) in one
call:

```
find_references(symbol="com.example.Calculator")
  → SYMBOL_RELOCATED
    "'com.example.Calculator' is not there any more — it appears to have moved
     or been renamed. Found: 'com.example.math.Calculator'."
    hint: STALE MEMORY: re-issue with 'com.example.math.Calculator' and
          remember THAT name from now on.
```

Renamed members get the same treatment (the type is still there, so its real
members are named, closest first). A name with nothing similar in the index gets
an honest `SYMBOL_NOT_FOUND` — *"it is gone, not moved"* — never a dressed-up
guess.

**The correction is never acted on.** The call still fails. Silently retargeting
a refactoring at a symbol you did not name would be far worse than a failed
call; you re-issue once with the corrected name, and your memory is now right.

## Search teaches the address

When a search lands an exact-name hit, the response tells you how to skip the
search next time:

> Address this directly next time: `symbol="com.example.Calculator"` — the name
> is the stable key (it survives file moves; a position does not), and every
> reading tool and whole-symbol refactoring accepts it.

A wildcard sweep teaches nothing — there is no single address to teach.

## Sessions start oriented

`inspect(kind=landmarks)` names the workspace's load-bearing types, ranked by
how much of the code leans on each — the head start a human has from having
worked in a codebase, instead of searching your way in every session. Cached per
workspace load.

Every landmark is verified addressable by its own name before being offered: a
landmark you cannot address would teach a name that fails.

## Fixes

- **`analyze(kind=type)` crashed on every binary type.** Asking about
  `java.util.ArrayList` — or any dependency class — returned an INTERNAL_ERROR
  in v2.10.0 and earlier: a class file has no compilation unit, and the member
  paths dereferenced it anyway. It now answers with the type's members and
  hierarchy (a binary has those; it just has no source lines). This is exactly
  the kind of type a memory-first agent addresses by name, so it mattered
  doubly.
