# GOJA MCP v2.2.6 — loader upgrades self-heal the corpus

Point release on v2.2.5.

## Fix: the skip-unchanged hash carries a loader fingerprint

Skip-unchanged (v2.2.1) made repeated loads free — and quietly blocked retroactive
enrichment: v2.2.5's name-as-symptom cues only reached files whose *bytes* changed, so
an existing corpus kept its old, cue-less entries forever (short of a destructive
reseed).

The stored hash is now `SHA-256(content + ";loader=N")` with an internal loader version
that bumps whenever ingest semantics change. Upgrading to such a release makes every
stored hash stale by definition — **the first `load` after updating re-ingests the whole
corpus once, automatically**, delivering v2.2.5's name cues to every entry with no user
action and no risk to recorded-only entries. Within one loader version, idempotency
remains byte-strict (`0 loaded, N unchanged`, zero writes).
