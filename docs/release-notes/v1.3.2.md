# goja-mcp v1.3.2 — `srp_cohesion` fluent-mutator fix (Sprint 20 follow-up)

A one-detector precision fix on top of v1.3.1, from re-dogfooding `srp_cohesion` against
GOJA's own source.

## `srp_cohesion` — exclude every fluent mutator, not just 2-statement setters
v1.3.1 excluded fluent builder setters of the exact shape `field = v; return this;`. But a
builder's *multi-field* / *multi-statement* setters — `scope(pkgs, syms)`, `source(kind, file)`,
`addEvidence(x)` (lazy-init then add) — all still `return this`, yet slipped past the
2-statement rule. Each touched a different field, so LCOM saw them as separate clusters and
kept flagging `SymbolFact.Builder` (down to 3 clusters, but still flagged).

The rule is now keyed on the fluent signature itself: **any method whose last statement is
`return this` is a fluent mutator and is excluded from the cohesion graph**, regardless of
length or how many fields it sets. A method that returns the field or a computation
(`a = a*2; return a;`) is still real work and still counts; `build()` (returns a *new* object,
not `this`) still counts.

Dogfood effect: `srp_cohesion` on GOJA's source drops 7 → 6, with the last builder false
positive gone. The remaining findings are genuine 2-cluster classes worth a human look.

## Unchanged from v1.3.1
`shotgun_surgery` (skips interfaces/abstract, threshold 10), `dip` (3), `isp` (0), `lsp` (0).

## Compatibility
Behaviour-narrowing for one kind; no API/tool-count change (still 39 tools, 30 kinds).
