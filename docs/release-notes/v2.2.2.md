# GOJA MCP v2.2.2 — shared-store maintenance hardening

Point release on v2.2.1 (Sprint 21b) — three fixes found by live use of the shared
store with multiple attached residents.

## Fixes

- **Loads no longer grow the store at all.** The automatic post-load refresh re-judged
  every symbol-bearing entry — including already-superseded ones — writing a redundant
  status UPDATE per entry on every load (~100 KB per click). Refresh now judges only
  ACTIVE (accepted/candidate) entries; a no-change load performs zero writes end to end.
- **Peers survive `compact`.** `compact` issues `SHUTDOWN COMPACT` on the shared
  AUTO_SERVER database, which closes it for EVERY attached resident — previously the
  other workspaces' residents were left with a dead connection (permanently
  "unreachable" in the studio). The store connection is now self-healing: validated
  before every statement, reconnecting transparently.
- **`wipe` compacts afterwards.** H2's MVStore never shrinks on deletes, so a wiped
  store kept its old file size. A wipe now ends with a compact — the file shrinks to an
  empty shell (schema only). Wipe deletes DATA, never the database file: the shared
  store, its schema and attached peers stay alive.
