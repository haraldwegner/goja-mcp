# GOJA MCP v2.2.1 — loads stop growing the store

Point release on v2.2.0 (Sprint 21b).

## Fix: `load` skips unchanged sources entirely

Re-running `load` used to rewrite every entry (delete + insert per source) — the entry
count stayed stable, but H2's MVStore appends until compacted, so the store file grew
by ~1 MB per load.

- The raw content of each ingested memory file is now SHA-256-hashed into its entry
  (**schema v3**, additive migration with the usual pre-migration backup).
- An unchanged source causes **no write at all** and is reported as `unchanged`
  (`files = loaded + unchanged`). Links of unchanged files are still followed, so an
  unchanged index still discovers new files.
- Changed or new files re-ingest exactly as before — `load` remains the "pick up
  what's new" action.
- Entries from stores written before v2.2.1 carry no hash yet: the first load rewrites
  them once, then the store is stable. Existing file bloat is reclaimed by one
  `experience(kind=compact)` (or the studio's *clean up*).
