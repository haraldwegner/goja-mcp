# goja-mcp v1.4.0 — Kerievsky "Refactoring to Patterns" (Sprint 19)

The **goal layer**: pattern-targeted refactorings, each a composition of behaviour-preserving
primitives whose target is a named pattern instantiation. One new tool, both directions.

## `refactor_to_pattern(kind, …)` — 8 transforms behind one front door
A single parametric apply tool (mirroring `extract(kind)`); tool count **39 → 40**. Every
transform applies by default with an `undoChangeId`, supports `auto_apply:false` where
meaningful, and produces **compiling** output (the multi-file transforms are verified by a real
compile check in their tests).

**Toward a pattern** (when complexity warrants):
- `compose_method` — extract a long method's disjoint sections into intention-revealing calls.
- `replace_type_code_with_class` — generate a type-safe enum from a group of `static final`
  type-code constants (introduces the abstraction; usage migration is the agent's call).
- `refactor_to_state` — a switch on an `int` state field → nested State classes + delegation.
- `refactor_to_command_dispatcher` — a type-coded action switch → nested Command classes selected
  by a `switch`-expression.
- `form_template_method` — two sibling methods sharing a skeleton → a template method in the
  abstract superclass + abstract steps for what differs.
- `refactor_to_visitor` — generate a `<Base>Visitor` interface + `accept` double-dispatch across
  an abstract hierarchy (the invasive boilerplate).

**Away from a pattern** (Kerievsky's distinguishing insight):
- `inline_singleton` — a GoF singleton whose uniqueness no longer matters becomes an ordinary
  class (call sites → `new Type()`, public ctor, holder + accessor stripped).
- `replace_pattern_with_idiom` — a pattern superseded by a language feature (v1.4: anonymous class
  implementing a functional interface → a lambda).

All transforms are **conservative**: they refuse shapes outside a safe, compiling envelope rather
than emit questionable code — the agent arbitrates *when* to apply; the tool is mechanical.

## Detection kinds (via `find_quality_issue`, family `kerievsky`)
New: `singleton`, `type_code`. Reused as triggers: `switch_statements`, `long_method`,
`speculative_generality`. The `family` enum gains `kerievsky` (enum now `quality`/`fowler`/`solid`/
`kerievsky`).

## The OCP cure
Sprint 20 ships no `ocp` detector (an OCP violation is meaningful only against the extension axis
you mean to keep open). v1.4 lands the **cure**: the churn traces `divergent_change` /
`shotgun_surgery` now point at the applicable `refactor_to_pattern` recipes
(`OcpCure.recipesFor(...)` — the seed of a future `get_target_recipe`).

## Infrastructure
Three composable change mechanisms now exist and are reused across the catalog:
`ChangeEngine.fromFileEdits` (multi-file edits), `RecipeEngine` (dependent apply-reparse with an
atomic composite undo), and `CreateCompilationUnitChange` (new files, undone by delete) — the
last two composing into one `CompositeChange` for transforms that both edit and add files.

## Compatibility
Additive: +1 tool, +2 detect kinds, +1 family tag. No breaking changes. 766 tests green.

## Known follow-ups
`RefactorToStateTool` still carries a few private helpers duplicated by `PatternSupport` (a
code-quality nit, deferred from the release cut); `refactor_to_visitor` / `replace_type_code`
introduce the abstraction but leave usage migration to the agent (conservative by design).
