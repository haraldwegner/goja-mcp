# goja-mcp v1.5.0 — Multi-step refactoring orchestration (Sprint 18, Track 1)

The **how** layer of the detect → goal → how → learn pipeline: the behaviour-preserving,
parity-gated engine that lets a *fallible* agent restructure Java without smuggling a
behaviour change past a green compile. **Zero net new tools** — every capability lands
behind an existing front door, so the surface stays at **40 tools**.

## Highlights

### The multi-step plan lifecycle — `refactoring(action=…)`
Four new actions on the existing `refactoring` tool:
- **`plan`** — decompose a kind into an ordered, inspectable step list (no changes made).
  Bootstrap kinds: `compose_method` (one `extract` step per section, ordered bottom-up),
  `replace_type_code_with_class`, `inline_singleton`, plus the OCP-cure recipes
  `refactor_to_state` / `refactor_to_command_dispatcher` / `form_template_method`.
- **`apply_plan`** — walk the plan step by step, **parity-gated after every step** (compile
  0/0 + a conservative purity check on the step's target method); on a step failure, a
  broken build, or a new-control-flow purity finding, **roll the whole plan back atomically**
  (reverse-order undo) and report. On success, compose the per-step undos into one plan undo
  handle and emit an `Outcome`.
- **`inspect_plan`** — review a plan: which steps are applied vs pending.
- **`undo_plan`** — revert a fully-applied plan via its composed undo.

### The parity + purity backbone (internal)
- **`ParityGate`** — build every project, clean iff no error markers remain.
- **`PurityCheck`** — a conservative AST diff flagging `NEW_CONTROL_FLOW_OUTCOME` (the
  `REFACTORING_LESSONS_LEARNED` `return null` case), `CHANGED_OUTBOUND_CALLS`, and
  `RELOCATED_SIDE_EFFECT`. Findings are surfaced; a new control-flow outcome halts the plan.

### The learning seam (internal)
- **`Outcome`** + **`Advisor`** (default `NoOpAdvisor`) — the orchestration loop emits an
  `Outcome` at cycle end and consults the advisor before planning. Sprint 21 fills the seam
  with the knowledge store; v1.5 ships it **stateless but store-aware**.

### Cost-tool primitives — the compiler writes the code, not the LLM
- **`generate(kind=copy_class)`** — clone the class at a caret into a new same-package file,
  renaming the type, its constructors, and every self-reference (binding-checked).
- **`extract(kind=superclass)`** — synthesize a new abstract parent and pull up the methods a
  class and its same-package siblings share (byte-identical + self-contained), as one atomic
  `CompositeChange`. Together they are the IDE-style copy → extract-superclass workflow, done
  by the compiler instead of re-authored token by token.

### OCP cure — detect → cure → execute closes
`divergent_change` / `shotgun_surgery` (the OCP trace) now point, via `OcpCure`, at a
**runnable** `refactoring(action=plan, kind=refactor_to_state | refactor_to_command_dispatcher
| form_template_method)` for a parity-gated run.

## Compatibility
No breaking changes. Every existing tool and action is unchanged; the additions are new
kinds/actions on `generate`, `extract`, and `refactoring`. Tool count remains **40**.
