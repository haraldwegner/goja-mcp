# GOJA MCP v2.2.4 — no more silent in-memory fallback on restart races

Point release on v2.2.3.

## Fix: transient lock contention is retried, not surrendered to

A runtime update restarts residents seconds apart. The dying resident's H2 lock file is
"recently modified", so H2 refuses the open — and the new resident silently degraded to
a **non-persistent in-memory store** while its peer took the file store moments later
(split-brain: half your loads went to RAM and vanished on the next restart; the Memory
view showed `in-memory` with no size).

The store open now retries transient lock messages ("Lock file recently modified",
"Database may be already in use") up to 5 times over ~7 seconds before any fallback.
Genuine failures — e.g. a store written by a newer resident — still fail fast, and the
in-memory degradation remains the last resort rather than the first response.
