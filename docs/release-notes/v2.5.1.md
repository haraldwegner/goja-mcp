# GOJA MCP v2.5.1 — workspace-scoped anchor judgment (hotfix)

Fixes a live incident found hours after v2.5.0: on the user-level SHARED store, a
resident's automatic post-load refresh judged **every** anchored entry against its own
workspace — a goja-workspace resident cannot resolve `pipeline.SlotManager` (an ORB
type), read that as "dead", and superseded 304 ORB entries during a routine memory
load. Two residents sharing the store would keep destroying each other's anchors on
every load.

## The fix

Anchor maintenance is now **workspace-scoped**:

- `refresh()` judges ONLY entries whose `workspace_id` provenance (stamped on every
  write since v2.1.0) matches the resident's own workspace; foreign and unstamped
  entries are skipped and reported (`foreign` count) — never superseded, never cleared.
- `backfillAutoAnchors()` is scoped the same way: a resident never anchors a foreign
  workspace's entry against its own types (common simple names could mis-anchor
  cross-workspace; absence beats a wrong pointer).
- A store WITHOUT a workspace identity (standalone, tests) keeps the previous
  judge-everything semantics.

## Notes

- Entries stamped before this release keep their stamps; a stamp only updates when the
  source re-ingests (content change or the next `LOADER_VERSION` bump re-stamps the
  md-loaded corpus).
- Deploying v2.5.x runtimes to residents sharing one store is SAFE from this release
  on — deploying v2.5.0 alone would have reintroduced anchor-flapping at scale.
