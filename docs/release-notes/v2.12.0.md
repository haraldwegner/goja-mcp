# jawata v2.12.0 — the debugger

**Sprint 24, phase 2.** jawata could already read your code the way the compiler reads
it. Now it can watch that code **run**.

One new front door — `debug` — and with it the thing no other Java MCP server has: an
agent can attach to a live JVM, stop it where it goes wrong, look at what it is actually
holding, ask it questions, change its mind, and let it go.

**44 tools** (was 43).

---

## What it does

**Attach, or launch.** `debug(action=discover)` lists the local JVMs and flags each one
`debuggable` or not. `attach` connects to a running one; `launch` starts a program under
the dev/sim preset — **held before its first instruction**, so you arm your breakpoints
and *then* start it. A target that is already running has already run past whatever you
wanted to see.

**Seven kinds of breakpoint** — line, method, conditional, hit-count, exception, and
field *access* and *write* as separate things. A class that is not loaded yet does not
fail: the breakpoint is **deferred** and says so, then binds the moment the class arrives.
"Not loaded yet" and "never going to bind" must not look the same.

**Look at the program.** `snapshot` gives the stack, the arguments, the locals and `this`,
expanded to bounds you set — and **every bound that bites is reported on the node it
truncated**. A silently truncated object graph reads like a complete one, and then you
conclude something false. A thread that is RUNNING reports no stack at all, and says why:
a stack read from a running thread is a race, not a reading.

**Ask it questions.** `evaluate` runs a Java expression *in the target*, including real
method calls on its own objects — `offset()` really executes over there. JDT parses the
expression (we host the best Java parser in the process); JDI executes it against the live
frame. An expression form the evaluator does not implement is **refused by name** rather
than quietly evaluated as something else.

**Change its mind.** `set_value`, `force_return`, `pop_frame`, `redefine` — test a
hypothesis on the running program. And because a program you have edited is no longer the
program you started, the session **records every change it made**: `status` always reports
`mutationCount` and `programIsUnmodified`. A session that cannot say what it changed cannot
be trusted to report what it found.

**Watch without stopping it.** `probe_set` — field watches, method traces (with the return
value), logpoints — read what the JVM's own events already carry and **suspend nothing**.
This is what you use on a live simulation, where stopping the world is exactly what you
cannot do. The one honest catch: *locals* can only be read from a stopped thread, so a
logpoint that captures expressions does stop it, briefly, and reports `perturbs: true`.
Calling that "non-suspending" would be a lie — and the worst kind, one that silently
shifts the timing of the race you are hunting.

**Find the bug.** `debug(action=replay)`: launch a program, declare what must always be
true, and stop at the **first** moment it is not. The invariant is armed as its own
negation, so the program runs at speed and stops only where it is genuinely broken. You get
the first violation — stack, locals, captured values — as a stored artifact, with the
thread **still standing in it**, so you investigate from the moment it broke rather than
backwards from the wreckage.

## The loop closes

A breakpoint hit names the class and the method. **That is the key the rest of jawata
already takes.** The hit hands you `symbol="com.example.Foo#bar"` pre-assembled, and it
goes straight into `get_call_hierarchy`, `find_references`, `analyze` — with no search in
between. A search would only re-derive what the running program has just told you, and could
get it wrong.

The same key works on the knowledge store: a lesson recorded against the symbol a breakpoint
produced is recalled by that symbol. What you learn at runtime is findable by the name the
runtime used.

## Being woken, rather than waiting

An MCP server cannot push anything into an agent — it only ever speaks when spoken to. But
your harness watches files. So every hit is appended, as one JSON line, to a per-session
`hitStream`: point a file monitor at it and **each breakpoint becomes a notification**. The
session stays live, the agent idles between hits, nothing is polled.

And nothing is ever missed. A hit *suspends the thread and holds it* — the program stops and
waits for you, indefinitely. A `wait` that times out loses nothing; come back an hour later
and it is still standing there. That is what a notification could never promise.

For a rare condition, hand the `sessionId` to a subagent and let it wait in its own context.

## What it costs you, said out loud

A conditional breakpoint stops the thread on **every** occurrence to evaluate the condition —
even the misses. So it reports `conditionEvaluations` and `passedOver`, and past 500 it warns
and points you at `hit_count`, which the JVM counts internally without stopping anything.
Cheapest first: a cold-path breakpoint → `hit_count` (free) → a condition (costly, exact).

## Read this before you point it at something

1. **A debug session suspends threads and changes state** in the JVM it attaches to. This is
   intervention, not observation.
2. **It is meant for a development or simulation machine** — and that is structural, not a
   policy we wrote. A JVM can only be debugged if it was *started* with a debug agent, and one
   cannot be added to a running JVM. A production JVM that was not launched for debugging is
   not refused; it is **unreachable**.
3. **Where a JVM *is* debuggable — a test or staging box — that is your professional
   judgment**, and jawata will not second-guess it. But if you do not know what suspending
   that JVM would do, that is the answer.

## Also in this release

**The "It has:" member list in a stale-symbol correction is no longer noise** (found in the
v2.11.1 dogfood, held back for this release). When you ask for a member that has moved or been
renamed, jawata answers with the members that *are* there — and that list used to repeat
overloads, offer the constructor as though it were a member you could call by name, and
truncate without saying so. It is now deduped, constructor-free, and bounded with a visible
`…`. A truncated list that looks complete is a small lie, and it produces a confidently wrong
next call.

**A long-standing family of rare, hard-to-reproduce wrong answers is fixed at its root.**
Under specific conditions — most notably two processes touching the same project directory
at once — a project could load with an incomplete classpath or an analysis could quietly skip
files it failed to read, and the answers computed on top looked normal. Project loading is now
safe against the interference we found, and — the bigger change — **no failure can silently
narrow a result anymore**: a load that cannot resolve its dependencies or mount its sources
fails with the cause; a scan that could not examine everything says exactly what it missed
(`filesListed` / `filesExamined` / `lookupFailures`); "none found" is only ever said about a
scan that actually looked at everything. Results you can act on without wondering whether a
re-run would say something else.

**Workspace hygiene:** projects removed or replaced via `load_project` are now fully released
(previously they could accumulate in a long-running server), and every refactoring refuses to
run against a workspace with an unreadable project — with a diagnosis naming the project, the
problem, and the remedy.

## Upgrading

Drop-in. No configuration changes, no new dependencies — the debugger is the JDK's own JDI
(`com.sun.jdi`), already in every JVM that runs jawata. Nothing you were doing behaves
differently; there is simply one more door.

Suite: 1311 tests, green — serial and sharded (three consecutive parallel runs).
