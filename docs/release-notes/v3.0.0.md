# jawata v3.0.0 — jawata drives: the agent-runner milestone

**The v3.0 milestone (Sprint 25).** Through v2.x, jawata was the compiler-accurate
toolbox a coding agent uses. From 3.0, **jawata drives its own narrow agents**:
jawata-studio 3.0.0 ships the runner — seat definitions, a pluggable model-CLI
adapter, a bounded detect→do→verify→stop loop, hard gates in front of every
change, containment ceilings with process-tree reaping, and a labeled run
journal — plus the first seat ladder (javadoc-writer, test-writer, architect,
debugger, profiler, spec-editor/spec-auditor), every one of them proven live
behind its gates. **Everything runs in propose mode: no seat lands a change
without a human yes.** The gates this side of jawata-mcp provides are the same
ones agents already know: `compile_workspace`, `run_tests` (coverage, delta,
mutation), the parity-gated `apply_plan`.

**Still 45 tools.** Drop-in. New capability arrives as actions on existing
doors, per the standing rule.

## New in jawata-mcp 3.0.0

- **Async family sweeps** — a `find_quality_issue` FAMILY sweep can take
  minutes and outlive a client's timeout; before, the server kept computing
  and the response was silently lost. Now `action=start` returns a `sweepId`
  immediately; `action=status` reports progress (`kindsDone`/`kindsTotal`)
  while running and returns the FULL result once finished — **retrievable
  repeatedly, so a timed-out client loses nothing**; `action=cancel` stops
  between detector kinds and `status` then returns the honest partial
  (`partial: true`). The synchronous path is unchanged.
- **organize_imports tells the truth** — its description and Javadoc now carry
  the known limit shipped (filed) in v2.14.1: adding missing imports fails
  loudly headless (a JDT import-rewrite defect, pre-existing); removal,
  static-import pruning and project-order sorting work and are tested. Use
  `quick_fix(action=suggest_imports)` for adds until the follow-up lands.

## The runner (jawata-studio 3.0.0) — what shipped behind it

- Seats are markdown definitions (model tier, effort, cron schedule, gate
  classes, ceilings, stance); model tiers are **evidence-set**: a measured
  table (same fixtures, same gates, small vs strong tier) decided each seat's
  tier, and the table ships in the sprint records.
- Proposals are whole-file blocks — the runner computes diffs
  deterministically (live runs falsified model-emitted unified diffs twice) —
  and every proposal record carries `proposal.md`, `diff.patch`, `gates.json`
  and the staged files.
- Gates judge the PROPOSED state on a shadow copy, never the live tree; a
  gate that cannot run has NOT passed; refusals keep their evidence.
- The architect seat has a **design mode**: target-architecture artifacts
  with the picture in them (`ARCHITECTURE-<scope>.md`), which its watch mode
  then judges checkpoint diffs against.
- The managed rule block deployed to clients now carries the
  **upward-communication contract** (decision-first asks, options with
  pros/cons, abbreviations defined at first use).

Full evidence per seat — live runs on the deployed binary with a real model
CLI, gate outcomes, costs — lives in the sprint-25 dossier.
