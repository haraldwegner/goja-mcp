# goja-mcp v1.3.1 — SOLID precision tuning (Sprint 20 follow-up)

A precision pass on the v1.3.0 SOLID detectors, driven by dogfooding them against GOJA's
own source. `dip` (3), `isp` (0), `lsp` (0) came back clean; two over-firing patterns are
fixed here.

## `srp_cohesion` — exclude fluent builder setters
A builder's `withX(v) { this.x = v; return this; }` methods each touch one field, so LCOM
saw them as disjoint clusters and flagged builders as "low cohesion" (the dogfood hit
`SymbolFact.Builder` with 6 false clusters). Fluent setters — a field assignment followed by
`return this` — are now excluded from the cohesion graph, alongside the getters/void-setters
already excluded. A method that returns the field or a computation (e.g. `a = a*2; return a;`)
is still real work and still counts.

## `shotgun_surgery` — skip abstractions, raise the threshold
The fan-in proxy flagged the most-depended-on **interfaces and abstract classes**
(`IJdtService`, `Detector`, `Tool`, `AbstractTool`…) as violations — but a widely-depended-on
abstraction is **DIP done right**, not shotgun surgery. So:
- **Interfaces and abstract classes are skipped** (only concrete types can be shotgun-surgery
  candidates).
- The default threshold is raised **5 → 10** distinct referencing types.

This drops the noise on `family="solid"` substantially; the concrete types that remain (e.g. a
value/DTO referenced by 100+ types) are genuine "a change here ripples" candidates.

## Compatibility
Behaviour-narrowing for two kinds; no API/tool-count change (still 39 tools, 30 kinds).
Pass a lower `threshold` to restore the v1.3.0 `shotgun_surgery` sensitivity.
